# Command Logger 스킬

## 목적
Claude Code 터미널용 명령어 파일을 만들 때, 로그 기록을 자동으로 함께 수행하여 "로그 까먹음" 문제를 구조적으로 방지한다.

## 핵심 원리
명령어 파일 생성과 로그 기록은 분리할 수 없는 하나의 행위다. 명령어 파일만 만들고 로그를 안 쓰면 다음 세션에서 맥락을 잃는다.

---

## 1. 세션 시작 시 (컨텍스트 복구)

새 세션이 시작되거나 컨텍스트가 끊긴 후 재개할 때, **어떤 작업이든 하기 전에** 아래 2개 파일을 반드시 읽는다:

```
1. docs/sessions/COWORK_CONVERSATION_LOG.md — 마지막 항목부터 역순으로 읽어 현재 상태 파악
2. docs/sessions/EXECUTION_STATUS.md — 명령어 파일별 실행 상태 (DONE/IN_PROGRESS/NOT_STARTED)
```

읽은 후 은태님에게 한 줄 요약을 보고한다:
> "마지막 작업: [내용]. 미실행 명령어: [N]개. 이어서 진행하겠습니다."

절대 로그를 안 읽고 즉흥으로 움직이지 않는다.

---

## 2. 명령어 파일 생성 시 (3단계 자동 실행)

명령어 파일을 만들 때 아래 3단계를 **하나의 작업으로** 수행한다. 1단계만 하고 2~3단계를 빼먹으면 안 된다.

### 2-1. 명령어 파일 생성
- 파일명: `CLAUDE_CODE_[내용].md`
- 위치: 프로젝트 루트 (`/Users/maegbug/potal/` 또는 현재 워크스페이스)

### 2-2. COWORK_CONVERSATION_LOG.md 업데이트
명령어 파일을 만든 직후, 아래 포맷으로 로그를 추가한다:

```markdown
### [HH:MM KST] [제목]
- 은태님 지시: [원문 요약]
- 결정: [무엇을 왜]
- 생성한 명령어 파일: **`파일명.md`**
- 내용 요약: [명령어 파일이 하는 일 1줄]
- 다음 단계: [이 파일을 어느 터미널에 넣을지]
```

### 2-3. EXECUTION_STATUS.md 업데이트
명령어 파일의 실행 상태를 기록/갱신한다:

```markdown
| 파일명 | 상태 | 터미널 | 시간 | 비고 |
|--------|------|--------|------|------|
| CLAUDE_CODE_FIX_17_AUTO.md | NOT_STARTED | - | - | 17개 순차 실행 |
```

상태값:
- `NOT_STARTED` — 생성됨, 아직 터미널에 안 넣음
- `IN_PROGRESS` — 터미널에서 실행 중
- `DONE` — 실행 완료, 결과 확인됨
- `FAILED` — 실행 중 에러, 재실행 필요

---

## 3. 실행 결과 수신 시

은태님이 터미널 결과 스크린샷을 공유하면:

1. 결과를 분석한다
2. COWORK_CONVERSATION_LOG.md에 결과 기록 추가
3. EXECUTION_STATUS.md에서 해당 파일 상태를 `DONE` 또는 `FAILED`로 갱신
4. 다음 작업을 제안한다

---

## 4. 세션 종료 시

세션이 끝나기 전에 반드시:

1. COWORK_CONVERSATION_LOG.md에 세션 요약 추가
2. EXECUTION_STATUS.md 최종 갱신
3. 미완료 작업이 있으면 "다음 세션에서 [X]부터 이어서" 명시

---

## 체크리스트 (매번 확인)

명령어 파일을 만들 때마다 스스로에게 물어본다:
- [ ] COWORK_CONVERSATION_LOG.md에 이 작업을 기록했는가?
- [ ] EXECUTION_STATUS.md에 이 파일을 추가했는가?
- [ ] 은태님 원래 지시를 정확히 기록했는가?
- [ ] 다음 단계가 명확히 적혀있는가?

하나라도 빠지면 명령어 파일 생성이 완료된 게 아니다.
