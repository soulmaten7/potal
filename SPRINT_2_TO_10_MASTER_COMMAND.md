## ⚠️ 절대 규칙 — 이 명령어의 모든 작업에 적용

1. **한 번에 1개 기능만** — 절대로 2개 이상 동시에 진행하지 않는다
2. **5회 검수 통과 후 다음으로** — 1개 기능 수정 완료 후 반드시 5번의 독립 검증을 거친다. 5회 전부 PASS여야 다음 기능으로 넘어간다
3. **직접 분석하고 직접 실행** — 각 기능에 대해 네가 직접 현재 코드를 읽고, 부족한 점을 분석하고, 수정하고, 검증한다
4. **엑셀 로깅 필수** — `POTAL_Claude_Code_Work_Log.xlsx`에 평소처럼 기록한다
5. **빌드 확인 필수** — 각 기능 완료 시 `npm run build` 통과 확인
6. **기존 코드 파괴 금지** — 다른 기능이 깨지지 않도록 주의
7. **추정 금지** — 실제 코드/DB를 확인하고 판단한다

## 🔄 각 기능별 실행 절차 (107개 모두 동일하게 적용)

### Phase 1: 현황 분석 (직접 분석)
```
1. 해당 F번호 기능의 관련 코드 파일을 전부 찾아서 읽는다
2. 현재 구현 수준을 판단한다:
   - API 엔드포인트가 있는가?
   - 비즈니스 로직이 완전한가?
   - 에러 핸들링이 있는가?
   - 타입 정의가 완전한가?
   - 테스트가 있는가?
3. "100% 완성"에 부족한 부분을 목록으로 정리한다
```

### Phase 2: 구현/수정
```
1. 부족한 부분을 하나씩 수정한다
2. 새 파일이 필요하면 생성한다
3. API 엔드포인트가 없으면 만든다
4. 비즈니스 로직이 불완전하면 완성한다
5. 에러 핸들링을 추가한다
6. TypeScript 타입을 완전하게 한다
7. 해당 기능의 테스트를 작성한다
```

### Phase 3: 5회 검수 (전부 PASS해야 다음으로)
```
검수 1: 빌드 검증
  - npm run build 실행
  - 에러 0개 확인

검수 2: 코드 완전성 검증
  - 해당 기능의 모든 파일을 다시 읽는다
  - API 엔드포인트 존재 + 정상 응답 확인
  - 비즈니스 로직 완전성 확인
  - 에러 핸들링 존재 확인
  - TypeScript 타입 에러 없음 확인

검수 3: 테스트 검증
  - 해당 기능의 테스트 실행
  - 전부 PASS 확인
  - 엣지 케이스 커버 확인

검수 4: 통합 검증
  - 다른 기능과의 연동이 깨지지 않았는지 확인
  - 관련 import/export 체크
  - 기존 테스트가 깨지지 않았는지 확인

검수 5: 최종 판정
  - 이 기능이 "프로덕션에 바로 배포해도 되는 수준"인지 판단
  - 부족한 점이 하나라도 있으면 Phase 2로 돌아간다
  - 100% 완벽하다고 판단되면 PASS
```

### Phase 4: 기록 및 다음 기능으로
```
1. 엑셀 로그에 기록 (POTAL_Claude_Code_Work_Log.xlsx)
   - 기능 ID, 기능명, 수정 내용, 생성/수정 파일, 테스트 결과, PASS 여부
2. 콘솔에 진행 상황 출력:
   "✅ [순서]/107 완료: F[번호] [기능명] — PASS"
3. 다음 기능으로 넘어간다
```
