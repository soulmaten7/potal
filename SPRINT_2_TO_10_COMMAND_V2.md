## 🚨 최우선 금지 규칙 — 반드시 읽고 지켜라

### 절대 금지 행위 (이것을 하면 작업 실패로 간주)
1. **"이미 구현됨" 판정 금지** — 코드 파일이 존재한다고 "이미 완료"로 넘어가지 마라. 모든 107개 기능은 미완성 상태다.
2. **"전수 확인" 판정 금지** — 파일을 훑어보고 "확인 완료"로 처리하지 마라.
3. **"인프라 완료" 판정 금지** — 인프라가 있어도 기능이 100%인 것은 아니다.
4. **여러 기능 동시 진행 금지** — 1개 끝나기 전에 다음 기능으로 넘어가지 마라.
5. **기능 묶어서 처리 금지** — "Sprint 3의 9개를 한번에 확인" 같은 방식 금지.
6. **10분 안에 전체 완료 불가** — 기능 1개당 최소 수분의 실제 작업이 필요하다.

### "완료"의 정의 (이 기준을 충족해야만 PASS)
기능 1개가 "완료"라고 판정하려면 아래 6가지가 **전부** 충족되어야 한다:
- ✅ 해당 기능 전용 API 엔드포인트가 존재하고, 실제 호출 시 정상 응답한다
- ✅ 비즈니스 로직이 완전하다 (stub, placeholder, TODO, "not implemented" 없음)
- ✅ 에러 핸들링이 있다 (try-catch, 400/404/500 응답, 입력 검증)
- ✅ TypeScript 타입이 완전하다 (any 타입 없음, 인터페이스 정의됨)
- ✅ 해당 기능 전용 테스트가 있고 PASS한다 (최소 3개 테스트 케이스)
- ✅ npm run build 통과한다

위 6가지 중 하나라도 빠지면 → 수정해서 충족시킨다. 넘어가지 않는다.

## 🔄 각 기능별 필수 실행 절차 (107개 전부 동일 적용)

### Step 1: 현황 분석 — 반드시 코드를 읽어라
```
1. 해당 기능 관련 파일을 find/grep으로 전부 찾는다
2. 찾은 파일을 전부 읽는다 (cat으로 내용 확인)
3. 아래 체크리스트를 하나씩 판정한다:

   □ API 엔드포인트가 있는가? → 있다면 URL과 method를 기록. 없으면 "없음"
   □ 비즈니스 로직이 완전한가? → stub/placeholder/TODO/"not implemented"가 있으면 "불완전"
   □ 에러 핸들링이 있는가? → try-catch, 입력 검증, 400/404/500 응답이 있는지 확인
   □ TypeScript 타입이 완전한가? → any 타입, 미정의 인터페이스가 있으면 "불완전"
   □ 전용 테스트가 있는가? → 테스트 파일 존재 여부 + 테스트 케이스 3개 이상 있는지 확인
   □ 실제 동작하는가? → 코드를 읽고 로직이 실제로 동작하는 코드인지 판단

4. 6개 중 부족한 항목을 "수정 필요 목록"으로 정리한다
5. 수정 필요 목록이 0개인 경우는 거의 없다. 0개라면 다시 꼼꼼히 확인한다.
```

### Step 2: 구현/수정 — 실제로 코드를 작성하라
```
수정 필요 목록의 각 항목을 하나씩 수정한다:

- API 엔드포인트 없음 → 새 route.ts 파일 생성, request/response 처리, 인증 체크 포함
- 비즈니스 로직 불완전 → stub을 실제 로직으로 교체, TODO 주석 제거
- 에러 핸들링 없음 → try-catch 추가, 입력 검증 추가, 적절한 HTTP 상태코드 반환
- TypeScript 타입 불완전 → 인터페이스 정의, any를 구체 타입으로 교체
- 전용 테스트 없음 → __tests__/ 아래에 테스트 파일 생성, 최소 3개 테스트 케이스
  (정상 케이스 1개 + 에러 케이스 1개 + 엣지 케이스 1개)
```

### Step 3: 5회 검수 — 전부 PASS해야 다음으로
```
검수 1: npm run build
  - 실행하고 "Compiled successfully" 확인
  - 에러가 있으면 수정 후 재실행

검수 2: 코드 완전성 재확인
  - Step 1의 체크리스트 6개를 다시 확인한다
  - 수정한 파일을 다시 읽어서 stub/TODO/any가 남아있지 않은지 확인

검수 3: 테스트 실행
  - 해당 기능의 테스트 파일을 실행한다
  - npx jest [테스트파일] --passWithNoTests 금지. 실제 테스트가 실행되어야 한다
  - 전부 PASS 확인

검수 4: 통합 영향 확인
  - 수정한 파일을 import하는 다른 파일이 깨지지 않았는지 확인
  - npm run build로 전체 빌드 재확인

검수 5: 최종 판정
  - "이 기능을 지금 고객에게 보여줘도 되는가?" 기준으로 판단
  - 하나라도 부족하면 Step 2로 돌아간다
  - 100% 확신이 있을 때만 PASS

⚠️ 5회 검수 중 1개라도 FAIL이면 → 수정 후 처음부터 다시 5회 검수
```

### Step 4: 기록 + 다음 기능
```
1. 엑셀 로그 기록 (POTAL_Claude_Code_Work_Log.xlsx)
2. 콘솔 출력: "✅ [N]/107 완료: F[번호] [기능명] — 수정 [N]건, 테스트 [N]개 PASS"
3. 다음 순서 기능의 Step 1로 넘어간다
```
